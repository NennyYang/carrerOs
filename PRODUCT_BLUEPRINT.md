# CareerOS: 个人职业运行舱

## 1. 作品定位

这是一个面向初级 AI 应用开发工程师的个人职业成长与抗风险系统。

它不应该变成一套每天催促你打卡、放大焦虑的任务管理器。它的作用是把一个人视为持续运行的系统，让你能够看到：

- 我已经积累了什么能力和证据。
- 市场现在希望一年经验的人具备什么。
- 我与更优岗位之间还差什么，以及哪些差距值得先补。
- 即使今天只做了一点点，我的系统是否仍然在积累能量。
- 如果出现裁员、部门解散或主动跳槽，我是否拥有可立即使用的求职资产。

一句话版本：

> 用低压力的日常输入，换取可见的能力增长、市场匹配度和职业安全感。

## 2. 真实问题拆解

### 2.1 技术变化带来的职业危机感

Codex、Claude Code 等工具使基础编码产出的价值迅速变化。系统不能只鼓励“多学技术”，而要帮助用户形成更不容易被替代的能力组合：

- 正确理解需求并拆解为可交付产品。
- 使用 AI 编程工具完成真实工程，而不是只生成 demo。
- 评估模型能力、成本、可靠性、安全性和用户体验。
- 构建可展示、可复用、可解释的项目证据。
- 持续识别下一段需要补齐的能力。

### 2.2 公司不稳定带来的就业风险

目标不是泛泛地“准备找工作”，而是持续维持一种就业就绪状态：

- 定期收集目标岗位 JD。
- 提炼限制一年经验岗位的共同技能栈。
- 计算个人能力与目标岗位的覆盖程度。
- 将学习成果转化为简历 bullet、项目案例和面试素材。
- 在需要求职时，快速得到可投递简历与岗位优先级建议。

### 2.3 工作消耗导致的学习阻力

系统必须承认：下班后和周末未必拥有足够的精力。它不以连续打卡、惩罚断档、强制目标为设计核心。

系统需要提供：

- 可选而非欠债式任务。
- 五分钟也算进步的微行动。
- 对休息、恢复、复盘同样可记录。
- 一段时间未操作时不显示失败或逾期堆积。
- 让用户打开系统时首先看到积累，而不是亏欠。

## 3. 核心体验隐喻：职业运行舱

首页是一个正在运行的个人系统，而不是待办列表。系统由几个彼此连接的模块组成，每个模块有状态、能量、输入和产出。

### 3.1 四个主模块

| 模块 | 含义 | 输入 | 可视化产出 |
| --- | --- | --- | --- |
| 技能引擎 | 已掌握和正在生长的能力 | 学习、项目实践、复盘、技术文章 | 能力图谱、证据强度、成长轨迹 |
| 市场雷达 | 外部招聘市场真实需求 | JD、职位标签、薪资区间 | 高频技能、目标岗位适配度、趋势 |
| 求职护盾 | 随时可拿出的就业资产 | 简历、作品、STAR 故事、面试题 | 就绪度、简历评分、缺失资产 |
| 精力舱 | 用户真实可承受的投入 | 今日状态、可用时间、完成感受 | 今日建议强度、节奏回顾 |

### 3.2 系统状态不叫“分数”

单一总分容易让用户觉得自己被判卷。建议首页呈现多个状态指标，不把人归结为一个数字：

- `系统运转中`：近 14 天存在任意有效输入。
- `能力储备`：有证据支持的技能数量及最近新增。
- `岗位覆盖`：针对选定岗位族的技能匹配比例。
- `求职就绪`：简历、作品、面试故事和岗位列表是否可用。
- `今日负载`：低、中、高，由用户精力选择决定，不由系统施压。

可以使用能量流、节点亮度、环形进度与时间线，但避免红色告警、连续天数断裂提示和落后排名。

## 4. 用户使用闭环

### 4.1 日常闭环：让上班空档或晚上变成可选积累

1. 用户打开首页，选择今天的状态：`只能做一点`、`普通能量`、`想深入推进`、`今天休息`。
2. 系统根据状态提供 1 至 3 个可选动作，每个动作显示预计耗时与产生的资产。
3. 用户完成任意动作后提交极简记录，例如链接、截图、代码仓库、三句话复盘或勾选确认。
4. 技能引擎中对应节点产生能量流，并说明这次输入为哪个岗位需求增加了证据。

动作示例：

| 状态 | 动作示例 | 时间 | 产出 |
| --- | --- | --- | --- |
| 只能做一点 | 读一段 RAG 评测说明，记录 1 个判断标准 | 5 分钟 | 知识碎片 |
| 只能做一点 | 修改一条项目简历描述 | 8 分钟 | 简历资产 |
| 普通能量 | 用 AI coding agent 完成一个项目小功能并记录验证方式 | 25 分钟 | 项目证据 |
| 想深入推进 | 实现一个模型评测或 agent workflow demo | 60-90 分钟 | 作品模块 |
| 今天休息 | 标记休息或写下今日消耗来源 | 1 分钟 | 精力记录 |

### 4.2 周度闭环：复盘系统而不是审判自己

每周生成一次运行报告：

- 本周增加了哪些可验证能力。
- 哪些动作真正产生了简历或作品资产。
- 目标岗位 JD 中新增了哪些高频要求。
- 下周只建议一个主要方向和若干可选微行动。
- 用户可以标记“这周很累，维持即可”，系统随之降低建议强度。

### 4.3 求职闭环：从 JD 到准备行动

1. 导入或粘贴一个 JD。
2. 系统结构化解析职责、必备技能、加分技能、年限、行业、薪资和地点。
3. 将 JD 与已有能力证据、简历内容、作品进行比对。
4. 输出匹配结论、证据不足项、最小补齐计划、简历针对性建议。
5. 将该岗位纳入市场雷达，帮助判断技能是否值得投入。

## 5. 功能框架

## 5.1 首页：运行舱总览

目的：用户每次打开，都能看到“系统还在运转，我有积累，也有下一步选择”。

组件：

- 能力系统图：技能节点、相关项目证据和与目标岗位连接的能量线路。
- 今日负载选择器：四档精力，不记录连续签到。
- 今日可选动作：最多三个，按当前负载排序。
- 近期积累时间线：显示学习、项目、JD 分析、简历升级等成果。
- 就业天气：目标岗位数量、覆盖率变化、需要关注的新技能。
- 紧急模式入口：公司发生风险时，一键进入求职冲刺视图。

## 5.2 技能引擎：能力图谱与证据库

技能不因为用户写下“我会 Python”而点亮，必须逐渐绑定证据。

技能层级建议：

| 级别 | 定义 | 证据例子 |
| --- | --- | --- |
| 了解 | 知道概念和适用场景 | 阅读笔记、简短解释 |
| 使用 | 可以在引导下完成任务 | 小练习、配置截图、代码片段 |
| 交付 | 在作品或工作中稳定使用 | 可运行项目、功能说明、测试 |
| 解释 | 可以说明选型和权衡 | 复盘、面试问答、技术文档 |
| 迁移 | 能应用到不同问题 | 多项目证据、复杂问题解决记录 |

初始技能域可围绕 AI 应用工程师组织：

- Python 工程：类型、测试、异步、API、依赖与部署。
- LLM 应用：Prompt、结构化输出、工具调用、流式交互、上下文管理。
- RAG 与检索：embedding、向量库、chunking、召回、重排、评测。
- Agent 工程：workflow、状态管理、工具安全、人机协同、可观测性。
- 模型评估与生产质量：评测集、准确性、延迟、成本、guardrail、监控。
- 数据与后端：SQL、缓存、队列、鉴权、日志。
- 工程化与交付：Git、Docker、CI/CD、云部署、文档。
- 产品与沟通：需求拆解、业务指标、演示、复盘、跨职能沟通。
- AI 辅助研发：任务分解、prompting coding agents、代码审查、验证与调试。

## 5.3 市场雷达：JD 分析与岗位地图

核心能力：

- 手动粘贴 JD 或导入已保存职位文本。
- 自动提取职位基础信息和技能标签。
- 合并同义技能，例如 `LangChain`、`LLM orchestration` 与 `Agent workflow` 的关联。
- 过滤目标样本，例如“一年经验或不限年限、薪资高于当前、AI 应用方向”。
- 生成岗位族画像：最常出现的技能、加分项、作品要求和行业方向。
- 对比个人技能证据，输出已覆盖、薄弱和缺口。

重要原则：

- 不因为一条 JD 提到某技术就安排学习；只有多个目标岗位出现或用户真正想投递时才提升优先级。
- 不以关键词命中冒充掌握；匹配结果必须区分“简历提到”与“存在项目证据”。

## 5.4 求职护盾：简历与面试资产

功能：

- 简历上传或结构化填写。
- 按目标岗位进行简历匹配评估。
- 标出空泛描述，建议补充行动、技术、结果与量化信息。
- 从日常成长记录中自动发现可转化为简历 bullet 的素材。
- 保存作品案例页：问题、方案、技术选型、AI 工具参与方式、验证结果、反思。
- 建立面试故事仓库：STAR 结构、常见追问、证据链接。
- 维护投递清单与岗位比较，不在 MVP 中自动投递。

简历评估不应只给分数，还需解释：

- 与该职位匹配最强的三个证据。
- 最可能在筛选中被质疑的三个缺口。
- 一周内最值得补上的一项资产。
- 哪些要求不值得为了单个岗位专门补齐。

## 5.5 精力舱：反压力设计

这是作品与普通学习打卡产品最不同的地方。

功能规则：

- 用户可以选择休息，休息不会降低能力状态。
- 未完成的可选动作当天结束即消失，不滚成债务。
- 首页优先展示历史积累和新证据，避免展示“连续未学习 N 天”。
- 用户可设置一周投入上限，例如工作日最多两次 25 分钟任务。
- 当用户连续反馈低精力时，系统推荐恢复类行为或简历整理类低认知动作，而非继续塞入新技术。
- 用户可以隐藏总览指标，只看行动与成果。

## 5.6 技术信号站：新技术推送

目的不是制造追新焦虑，而是让用户知道与目标岗位真正有关的变化。

推送类型：

- 与当前目标 JD 高频技能相关的新工具、新框架和实践变化。
- 官方发布的模型、API、平台能力更新。
- 适合转化为作品特性的技术主题。
- 已学技能的更新，例如框架 breaking changes 或主流替代方案。

每条推送需要包含：

- 发生了什么。
- 为什么与你有关，关联哪些目标岗位或技能节点。
- 是否值得现在学：`忽略`、`观察`、`值得试用`、`应补齐`。
- 一个不超过 30 分钟的验证动作。

## 6. 关键指标与计算逻辑

系统指标的用途是帮助判断下一步，而不是评价人的价值。

### 6.1 技能证据强度

每个技能节点按照证据类型累积可信度：

| 证据 | 权重示例 |
| --- | --- |
| 阅读/笔记 | 1 |
| 小实验 | 2 |
| 可运行功能 | 4 |
| 有评测或测试的作品功能 | 6 |
| 能解释权衡的案例总结 | 4 |
| 工作或开源真实使用 | 8 |

可以设置衰减提示：超过一定时间未使用的技能不是“消失”，而是提示适合复习或通过项目重新激活。

### 6.2 岗位覆盖度

岗位覆盖度需分别展示：

- 关键词覆盖：你是否提及该技术。
- 证据覆盖：是否有东西证明会用。
- 深度覆盖：证据强度是否达到该岗位期望。
- 资产覆盖：简历、作品和面试叙事是否已表达出来。

这样能避免出现“学了很多，但招聘方看不见”的问题。

### 6.3 求职就绪度

建议展示为清单状态而非单个百分制：

- 有一份基础简历。
- 有一份面向目标岗位调整的简历。
- 有两个以上可以演示的 AI 应用项目案例。
- 有五个可复述的面试故事。
- 有当前目标岗位列表与优先级。
- 已覆盖目标岗位最关键的若干技能证据。

### 6.4 成就感来源

系统应该奖励资产化行为，而不只是耗时：

- 新增了一个可信技能证据。
- 一个学习片段变成了作品功能。
- 一个作品结果变成了简历描述。
- 一条简历描述匹配到了更多目标岗位。
- 一次复盘减少了无效学习方向。

## 7. 建议的信息架构

主导航保持简单：

| 页面 | 作用 |
| --- | --- |
| 运行舱 | 首页总览、今日动作、能量流与近期成果 |
| 能力图谱 | 技能节点、证据、成长路径 |
| 市场雷达 | JD 库、岗位画像、技能趋势与匹配 |
| 求职护盾 | 简历、作品案例、面试素材与投递准备 |
| 日志 | 每日微记录和周度复盘 |
| 设置 | 职业目标、当前薪资基线、时间与负载偏好、隐私配置 |

不建议第一版就做社交、排行、公开展示、连续签到、复杂日历排期。

## 8. 数据对象设计

第一版需要保存的核心数据：

| 对象 | 主要字段 |
| --- | --- |
| Profile | 当前角色、经验年限、薪资目标、目标岗位族、城市/远程偏好 |
| Skill | 名称、领域、层级、目标层级、证据强度、关联 JD |
| Evidence | 标题、类型、日期、技能、链接/文本、可用于简历与否 |
| JobDescription | 公司、职位、薪资、年限、文本、技能标签、优先级 |
| Resume | 版本、目标岗位、内容、匹配结果、改进建议 |
| PortfolioCase | 项目背景、解决方案、技术、结果、证据、面试叙事 |
| DailyLog | 日期、精力状态、选中动作、成果、感受 |
| WeeklyReview | 一周新增资产、市场变化、下周方向、节奏选择 |
| TechSignal | 来源、主题、关联技能、相关度判断、试用动作 |

## 9. AI 功能边界

适合使用大模型的部分：

- 从 JD 中提取结构化字段和技能标签。
- 对多条 JD 归类、总结共同要求。
- 分析简历和 JD 的匹配关系并给改写建议。
- 将完成记录转化为作品案例草稿或 STAR 故事。
- 根据精力状态生成低压力行动候选。
- 总结技术更新与用户当前目标的关联。

不应让模型独断的部分：

- 判定一个人是否“合格”或是否应该辞职。
- 夸大岗位匹配程度。
- 未经用户确认直接改写最终简历或对外提交。
- 将没有证据的技能包装为掌握。
- 因为热点新闻强行改变用户学习路线。

## 10. MVP：第一版只做什么

第一版目标：让用户在一周内开始使用，并能第一次看到能力与岗位之间的连接。

### MVP 必做

1. 运行舱首页
   - 今日精力选择。
   - 三个可选微行动。
   - 技能节点与近期成果的可视化状态。

2. 能力图谱和证据记录
   - 预置 AI 应用工程师技能分类。
   - 手动新增证据。
   - 技能证据强度及层级显示。

3. JD 分析
   - 粘贴 JD 文本。
   - 提取技能标签和年限/薪资等字段。
   - 对比当前证据，形成缺口列表。

4. 简历检查基础版
   - 粘贴简历文本。
   - 对某条 JD 输出匹配项、缺失项和可改进 bullet。

5. 轻量记录与周报
   - 每天可以记录完成成果或选择休息。
   - 展示一周新增证据和下一项推荐行动。

### MVP 暂缓

- 自动爬取招聘网站职位。
- 自动定时外部推送。
- 自动投递职位。
- 多平台登录、公开分享和社交功能。
- 高精度薪资统计与行业分析。
- 复杂知识库和全文检索。

暂缓原因：这些功能涉及平台合规、外部数据可靠性、成本和隐私，且不影响第一版验证核心体验。

## 11. 实现路径建议

当前项目是一个空白 Python 工程，适合先快速构建本地优先版本，以便个人真实使用。

### 11.1 推荐技术栈

| 层 | 第一版推荐 | 原因 |
| --- | --- | --- |
| 应用框架 | Streamlit | Python 起步快，适合迭代信息面板和输入分析 |
| 本地数据库 | SQLite + SQLModel | 数据结构明确，个人隐私数据本地保存 |
| 图表与图谱 | Plotly / Pyvis | 快速完成趋势和技能节点可视化 |
| AI 接口 | 可配置 LLM provider | 后续可以接 OpenAI 或其他兼容接口 |
| 解析模式 | Pydantic 结构化输出 | JD 与简历分析结果可验证可保存 |
| 配置 | `.env` + 设置页 | 不在代码中保存密钥 |

若未来希望作品更像成熟产品，可在数据与流程稳定后迁移为：

- 前端：React / Next.js。
- 后端：FastAPI。
- 数据库：PostgreSQL。
- 调度任务：Celery / APScheduler。

### 11.2 第一轮开发顺序

1. 建立基础 UI、数据库模型与种子技能树。
2. 构建运行舱首页，先使用模拟数据把可视化体验做对。
3. 加入证据录入和技能节点升级规则。
4. 加入 JD 粘贴、结构化解析和本地保存。
5. 加入简历文本与 JD 对比。
6. 加入每日轻记录和周度复盘视图。
7. 最后再接真实大模型和技术信号来源。

## 12. 容易遗漏但非常重要的部分

### 12.1 “学习”必须转化成外部可见证据

学会技术只是内部感受，跳槽时真正有价值的是对方可验证的表达。每条学习路径都应尽量走向下列至少一个产出：

- GitHub 项目功能。
- 可演示 demo。
- 评测结果。
- 架构说明。
- 简历 bullet。
- 面试故事。

### 12.2 AI 工具时代的作品评价标准要变

项目不能只展示“我写了代码”。更重要的是说明：

- 为什么选择这个问题。
- 你如何让 coding agent 帮助开发。
- 你如何验证生成结果不是表面可用。
- 你如何处理成本、失败、准确性和用户体验。
- 你最终独立作出的工程判断是什么。

这个作品本身就可以成为一项最强证据：你在构建一套由 AI 辅助、但由你负责判断与验证的真实个人产品。

### 12.3 需要有紧急求职模式

平时系统以低压力积累为主。一旦发生公司风险，用户可以切换到紧急模式：

- 立即整理当前可投岗位。
- 生成简历版本准备清单。
- 找出一周可补齐的关键缺口。
- 集中展示可讲的项目和面试故事。
- 跟踪投递和面试反馈。

### 12.4 需要保护隐私和心理边界

- 简历、薪资和工作记录默认仅本地保存。
- 外发到模型前明确告知哪些文本会被发送。
- 允许脱敏公司与个人信息。
- 不使用羞耻式提醒和断签惩罚。
- 不以系统建议替代现实中的职业、心理或劳动权益支持。

## 13. 一个可落地的首月使用剧本

### 第 1 周：建立底盘

- 填入个人目标和当前技能自评。
- 导入 5 至 10 条真正想投的 JD。
- 粘贴当前简历。
- 记录两个现有项目或工作案例。
- 系统输出第一个岗位覆盖视图。

### 第 2 周：生成证据

- 挑一个高频但证据不足的技能。
- 用两到三个小动作增加一个可演示功能或技术说明。
- 将成果转换成简历 bullet 与面试故事。

### 第 3 周：校正市场方向

- 新增一批目标 JD。
- 比较技能需求是否稳定。
- 删除只为少数职位出现的无效学习方向。

### 第 4 周：模拟风险发生

- 开启紧急求职模式。
- 判断现在是否有简历可发、有作品可讲、有岗位可选。
- 从模拟结果决定下一月最值得积累的资产。

## 14. 视觉方向

主题不是“赛博焦虑”或游戏化升级，而是一个安静、可信、有生命力的个人控制室。

视觉建议：

- 首页中心为个人系统拓扑图，四个模块由数据流连接。
- 技能节点被证据点亮，成长时出现轻微能量流动和新增标记。
- 市场需求作为外围信号进入系统，与已有节点形成连接或提示空位。
- 今日行动位于侧栏或底部操作带，保持清晰易选。
- 使用克制的多色状态：青绿表示稳定积累，琥珀表示待关注，蓝色表示市场信号，中性色表示休息或暂不投入。
- 避免红色倒计时、连续签到火焰、排行榜和压迫性的巨大百分比分数。

## 15. 作品名称候选

| 名称 | 气质 |
| --- | --- |
| CareerOS / 职业运行舱 | 直接、产品化、易解释 |
| Capability Engine / 能力引擎 | 强调长期积累与技能运转 |
| Northstar Console / 北辰控制台 | 强调方向和应对变化 |
| Workproof / 职业护盾 | 强调抗风险与求职准备 |

首选建议：`CareerOS - 职业运行舱`。它足够清楚，也能承载系统可视化、市场雷达与求职护盾的全部功能。

## 16. 下一步开发决策

在进入编码前，只需要确认三个产品选择：

1. 第一版是完全本地私用，还是将来希望公开部署作为作品展示。
2. 第一版是否立即接入真实 LLM API，还是先使用模拟分析结果完成交互体验。
3. 首页视觉更偏控制台信息面板，还是更强调“能量流动系统”的沉浸可视化。

无论如何，最适合的第一个可运行版本都是：

> 本地运行的 CareerOS 首页 + 技能证据库 + JD 对比功能，以你自己的真实数据作为第一位用户进行迭代。

## 17. 首轮原型反馈后的调整

### 17.1 能力图谱改为项目模板解析

用户不应被要求频繁手工提交零散“证据”。第一版应优先提供结构化项目档案：

- 做了什么项目。
- 使用了什么技术栈。
- 解决了什么具体问题。
- 采用什么方案解决。
- 如何验证结果，是否存在演示、评测或代码产物。

系统基于这些内容推导能力节点与强度，并要求关键判断能够回溯至项目事实。后端阶段可使用模型辅助抽取，但评分必须保留证据约束，不能仅依据自述自动抬高能力。

### 17.2 市场雷达改为岗位集合扫描

市场雷达的主要入口不再是逐条粘贴 JD，而是：

- 岗位名称。
- 目标城市。
- 发布时间范围。

系统获取一组满足条件的岗位后，中心结果应输出岗位集合中的共性门槛，例如 `Docker 部署为多数目标岗位要求`、`RAG 评测正在成为高频能力`，再展示单条职位和个人覆盖情况。

数据接入必须遵守招聘平台规则。以 BOSS 直聘为例，其用户协议禁止通过蜘蛛、爬虫或拟人程序等非正常浏览方式获取数据，因此正式版不将未经授权的自动抓取作为默认方案。可探索授权数据接口、允许聚合的数据源，或由用户主动提供合法获得的数据。

### 17.3 求职护盾聚焦简历过筛

能力判断由技能引擎负责；求职护盾不重复分析“你会什么”，而是聚焦：

- 某份简历与某条目标 JD 的初筛对齐。
- 招聘方在简历中已经能看到的匹配优势。
- 明明具备但尚未表达出来的能力。
- 为通过初筛最值得补写的项目 bullet 与关键词。

### 17.4 运行舱需展示持续运转和状态保存

系统拓扑不能只有静态节点，需要展示正在发生的处理过程，例如项目能力映射、岗位信号聚合、简历过筛检查和数据流变化。

第一版原型使用浏览器本地存档保留用户操作结果；进入后端阶段后增加真实用户账户、登录以及跨设备同步。
